学习日记20.01.21

这两天我对filter与listener有了一些了解，通俗的讲：

  • Listener就是监听器，它可以对一些对象进行监听，监听对象的启动与销毁，存入与取出。同时还能监听对象是否与session绑定与解绑，以及对象是否被钝化活化。它可以用于在servlet启动时，插入一些需要的代码，以达成需要的结果。

  • Filter就是过滤器，他可以针对用户的相应的配置，过滤请求。它在请求的资源符合设定是拦截请求，拦截后可以插入任意代码，可以修改请求内容，最后决定是否放行请求。

Listener

今天学习的实例，是通过Listener监听服务器启动，在服务器启动时打开定时任务。

  1. 首先创建自定义监听器，实现ServletContextListener接口

  2. 在web.xml中配置监听器。

    <listener>
<listener-class>com.xxxx.listener.BirthdayListener</listener-class>
</listener>

  3. 实现生日定时发送功能

首先，我们要分析生日提醒的业务。

当服务器启动时，设置一个定时任务，指定为今天的某个时间点，时间间隔为一天。这样的话，只要服务器启动，就会每天定时执行一次。

在定时任务中，我们需要首先找出当天生日的所有人。用户的生日在数据库中一般会使用date格式进行保存，例如：2019-07-10。

我们通过MM-dd的格式，通过sql模糊查询，能查到今天生日的所有用户。

然后循环进行邮件发送就好。

代码实现如下：

package com.xxxx.listener;

import java.sql.SQLException;
import java.text.ParseException;
import java.text.SimpleDateFormat;
import java.util.Date;
import java.util.List;
import java.util.Timer;
import java.util.TimerTask;

import javax.mail.MessagingException;
import javax.servlet.ServletContextEvent;
import javax.servlet.ServletContextListener;

import org.apache.commons.dbutils.QueryRunner;
import org.apache.commons.dbutils.handlers.BeanListHandler;

import com.ilshmos.domain.User;
import com.ilshmos.utils.DBUtils;
import com.ilshmos.utils.MailUtils;

public class BirthdayListener implements ServletContextListener {

    @Override
    public void contextDestroyed(ServletContextEvent arg0) {
        //什么也不做
    }

    @Override
    public void contextInitialized(ServletContextEvent arg0) {
        //监听器监听到服务器启动，则开始进行生日推送的任务
        SimpleDateFormat fullFmt = new SimpleDateFormat("yy-MM-dd hh:mm:ss");
        SimpleDateFormat yMdFmt = new SimpleDateFormat("yy-MM-dd");
        SimpleDateFormat MdFmt = new SimpleDateFormat("MM-dd");
        Date today = new Date();
        Date firstTime = null;
        try {
            firstTime = fullFmt.parse(yMdFmt.format(today) + " 08:00:00");
        } catch (ParseException e) {
            e.printStackTrace();
        }
        Timer timer = new Timer();
        timer.scheduleAtFixedRate(new TimerTask() {
            @Override
            public void run() {
                //这里定时发邮件
                //1、找出当天生日的所有人
                QueryRunner qr = new QueryRunner(DBUtils.getDataSource());
                String sql = "select * from user where ubirthday like ?";
                List<User> ulist = null;
                try {
                    ulist = qr.query(sql, new BeanListHandler<User>(User.class), "%" + MdFmt.format(today));
                } catch (SQLException e) {
                    e.printStackTrace();
                }
                //2、循环创建信息并发送邮件
                for (User user : ulist) {
                    String emailMsg = "亲爱的：" + user.getUsername() + ",今天是您的生日，祝您生日快乐！";
                    try {
                        MailUtils.sendMail(user.getUemail(), "生日快乐", emailMsg);
                    } catch (MessagingException e) {
                        e.printStackTrace();
                    }
                }
            }
        }, firstTime, 1000*60*60*24);
    }

}

以上代码有一句错误，这也是我写博客刚发现，因为我最初版的代码非常混乱。随处都是日期的调用，后来我将所有需要用到的SimpleDateFormat都给整理了一下，在整理的过程中欠缺考虑，造成了那个错误。

错误就是，在定时任务执行到查询当天生日的所有人时，它使用的是qr.query(sql, new BeanListHandler<User>(User.class), "%" + MdFmt.format(today));。本来这里的语句是new Date()，我当时在优化时，想到有些数据会被重复调用，干脆就写一个方法内的局部变量供方法内任意地方都可调用。但是，我欠缺考虑的是，第一个需要使用new Date()的地方，是在服务器启动时，设置定时任务，设置定时任务的初次执行时间。这里在方法开始声明无可厚非，然而在定时任务内部，需要在每天都获取到当天的生日，再用之前声明好的变量就不行了。

因为之前声明的变量指的是服务器初次启动的那天，而在定时任务内部使用，它就会每天都向“服务器启动时间“那天生日的人发邮件。

这里应该改为new Date()，代表每执行一次定时任务，就获取执行定时任务的当天日期。

Filter

过滤器有时候可以用于将共用代码抽离。比如，自动登录功能，字符编码功能。

自动登录

我在很早之前，也做过自动登录相关的东西，原理就是在每个页面都写上同样的功能，如果cookie里存有用户凭据。被访问的页面就执行登录功能，并将信息存入session。

但是..这种需要在每个页面中都写入相应的功能，当初我还没将页面的header等分层，所以更麻烦。

即使分层了，也不能保证在访问任意资源时都能自动登录。

因此，如果通过监听器。拦截访问WEB应用下任意资源的请求，在此刻读取cookie进行登录，就简单的多了，代码也只需要写一遍。

需求分析：

  1. 用户在登录页面登录，如果用户勾选上“自动登录”复选框，则在登录时，将用户名与密码存入cookie中。(实际开发应该是token之类的)

  2. 创建过滤器，过滤器内首先判断session中是否有用户数据。如果有代表用户的登录状态存在，就不执行代码直接放行；如果没有，则代表用户未登录，就从cookie中获取用户名和密码，如果获取到不为Null，就执行登录。如果没有获取到，则直接放行…

  3. 配置web.xml

代码实现：

<!-- web.xml配置 -->
<filter>
    <display-name>AutoLoginFilter</display-name>
    <filter-name>AutoLoginFilter</filter-name>
    <filter-class>com.xxxx.web.filter.AutoLoginFilter</filter-class>
  </filter>
  <filter-mapping>
    <filter-name>AutoLoginFilter</filter-name>
    <url-pattern>/*</url-pattern>
  </filter-mapping>
//过滤器中doFilter方法
public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) throws IOException, ServletException {
        HttpServletRequest req = (HttpServletRequest) request;
        Cookie[] cookies = req.getCookies();
        HttpSession session = req.getSession();
        if(session.getAttribute("user") != null) {
            chain.doFilter(req, response);
            return;
        }
        String username = null;
        String password = null;
        if(cookies != null) {
            for (Cookie cookie : cookies) {
                if("username".equals(cookie.getName())) {
                    //防止中文用户名，存入cookie会报错。
                    //对其进行编码解码可避免中文字符存入cookie报错
                    username = URLDecoder.decode(cookie.getValue(),"UTF-8");
                }
                if("password".equals(cookie.getName())) {
                    password = cookie.getValue();
                }
            }
        }
        if(username != null && password != null) {
            UserService service = new UserService();
            User user = null;
            try {
                user = service.login(username,password);
            } catch (SQLException e) {
                e.printStackTrace();
            }
            session.setAttribute("user", user);
            chain.doFilter(req, response);
            return;
        }
        chain.doFilter(req, response);
    }

字符编码

通常的，我们在servlet与前端进行数据传输时，如果遇到中文，就需要针对着进行编码处理。

response在写中文时，可以提前写一句response.setContentType("text/html;CharSet=UTF-8");

这样就可以解决response在传输过程中的中文问题。

而request则不一样，它在不同的提交方式下，需要进行不同方法的处理。在post提交方式下，我们只需要写request.setCharacterEncoding("UTF-8");即可。但是如果在get提交方式下，因为请求内容在请求行，所以前面那句代码就不起作用了，我们只能以最原始的方法，对获取到的参数进行重新编码。

而对参数进行重新编码这种方法，同时也适用于post方式提交。

基于此，我们可以对request.getParameter()这个方法进行增强。

增强某个方法的时候，我们可以使用装饰者模式。原(request)对象与我们增强的类继承同一个类，然后对我们的类中的相应的方法进行重写，达到增强的目的。

比较好的是，tomcat已经为我们提供了HttpServletRequestWrapper这个类，供我们对request中的方法进行增强。

代码实现：

package com.xxxx.web.filter;

import java.io.IOException;
import java.io.UnsupportedEncodingException;

import javax.servlet.Filter;
import javax.servlet.FilterChain;
import javax.servlet.FilterConfig;
import javax.servlet.ServletException;
import javax.servlet.ServletRequest;
import javax.servlet.ServletResponse;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletRequestWrapper;

public class EnhanceRequestFilter implements Filter {

    public void destroy() {
    }
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) throws IOException, ServletException {
        HttpServletRequest req = (HttpServletRequest) request;
        /*
         * 此处的纰漏是未重写全部的获取参数方法，比如getParameterValues等
         */

        EnhanceRequest enhanceReq = new EnhanceRequest(req);
        chain.doFilter(enhanceReq, response);
    }
    public void init(FilterConfig fConfig) throws ServletException {
    }

}
class EnhanceRequest extends HttpServletRequestWrapper{
    public EnhanceRequest(HttpServletRequest request) {
        super(request);
    }
    @Override
    public String getParameter(String name) {
        String value = super.getParameter(name);
        if(value != null) {
            //不勾选自动登录时，获取的参数为空，直接进行对象调用会报空指针异常
            try {
                value = new String(value.getBytes("ISO8859-1"),"UTF-8");
            } catch (UnsupportedEncodingException e) {
                e.printStackTrace();
            }
        }
        return value;
    }

}
<filter>
    <display-name>EnhanceRequestFilter</display-name>
    <filter-name>EnhanceRequestFilter</filter-name>
    <filter-class>com.xxxx.web.filter.EnhanceRequestFilter</filter-class>
</filter>
<filter-mapping>
    <filter-name>EnhanceRequestFilter</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>

在配置xml时，需要注意的是，filter执行的顺序。它执行的顺序按照在xml中“filter-mapping”的顺序进行执行。所以我们应该将这个增强功能的过滤器放在上面，这样才能使所有(包括下一个过滤器中的)请求中的request都进行了增强。

框架

我们在此其实不需要了解的过于深，过于透彻。

因为这种过滤器和监听器一般用于编写框架才会使用，例如Spring与Struct2等，都相应的用到了过滤器与监听器。
