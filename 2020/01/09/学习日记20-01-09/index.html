学习日记20.01.09

今天主要分页相关的内容进行了一次补充，因为我知道“分页”功能的原理，但是在真的对该功能进行实现的过程中，还是需要对”原理“进行补充的。

条件查询

  • 需求：对商品进行条件查询，条件有：搜索框（商品名）、下拉框（是否热门）、下拉框（类别）

实现的过程中，其实只需要注意dao层的sql语句的拼接就好。
因为三个条件是否存在都不清楚，所以程序必须有判断条件是否存在的能力。
当三个条件都没填写的时候，点击“搜索按钮”将列出所有数据，而填写了其中任意一个、两个或三个条件的时候，就按照对应的条件查找。
那么在实现过程中，如何拼接sql语句呢？分析一下就知道，平常的拼接方法肯定是不成的。
例如：

//这是错误做法
StringBuffer sql = new StringBuffer("select * from product");
if(condition.getPname() != null && !"".equals(condition.getPname().trim())) {
            sql.append(" where pname like ? ");
            params.add("%" + condition.getPname() + "%");
        }
if(condition.getIs_hot() != null && !"".equals(condition.getIs_hot())) {
            sql.append(" and is_hot = ? ");
            params.add(condition.getIs_hot());
        }
if(condition.getCid() != null && !"".equals(condition.getCid())) {
            sql.append(" and cid = ? ");
            params.add(condition.getCid());
        }

上述代码是错误的，我们思考一下，假设我们只想根据”是否热门“这个条件进行查询，那么根据上述代码，sql语句会变成什么呢？select * from product and is_hot = ?，这很明显是一句错误的sql语句。
其实只需要一点小改变就能实现需求，首先肯定要把其中的 where pname like ?改成 and pname like ?。三种条件都是有可能添加，直接追加为and是最好的方法，但是如果这样的话，我们就需要把sql语句改为select * from product where，这么做的话，如果我们填写了任意一种或多种条件，程序都可以正常执行，但是如果我们三个条件都没用，那么sql语句就会变成select * from product where，它单独作为sql语句肯定也是错误的。因此，我们还需要做一些改变。

为了使上面的sql语句能够单独成句，但是又不影响后面追加的条件。我们可以直接添加一个没有任何影响的条件，将sql语句改为select * from product where 1=1，这样的话它既能单独成句，也不会影响后续追加条件。而且查询结果也与需求或者说预想的一致。

需要注意的是！多个字符串拼接时，一定要注意sql语句中的空格。比如sql.append("and pname like ? ");如果这句代码是这样，那么最终在执行拼接时得到的sql语句就会是select * from product where 1=1and pname like ?，这样的sql语句明显错误。

分页

分页就是前端点击任意一页，后端会传来该页相应的数据。原理是通过sql的limit语句对查询结果进行限制，分页的好处是一次不会传输所有的数据，以防前端接受的数据过于庞大造成运行迟缓甚至崩溃。

我虽然知道分页的原理，但是从来没彻底的去实现过，今天在学习的过程中还是收获了很多。

首先我们需要将分页查询需要的数据包装成bean。

public class PageBean<T> {
    private long currentPage;//当前页
    private long totalPage;//总页数
    private long currentPageCount;//当前页显示多少数据
    private long totalProductCount;//总共多少数据
    private List<T> productList = new ArrayList<T>();
    //......后续省略
    }

  • “当前页”我们可以通过前端得到

  • 当前页显示多少数据可以根据前端计算出，也可以规定好

  • 总共多少数据我们可以通过dao层对数据库进行查询得到

  • 总页数其实就是Math.ceil(1.0*totalProductCount() / currentPageCount)，其中添加的”1.0”其实是为了防止整型相除最后得到的整型。比如我们有47条数据，每页显示12条数据。那么总页数就是(47/12)取整(直接去掉小数部分)加1，即4页；

  • 而查询到的List集合我们通过dao层进行查询得到。

  • 在查询前，我们需要首先获得limit的两个参数。第一个参数index就是(当前页数-1) * 偏移，第二个参数就是偏移，也就是当前页显示多少数据。例如每页显示12条，第一页的index为(1-1)*12 = 0，第一页的偏移为12；第二页的index为(2-1) * 12 = 12，偏移为12。

为什么要再pageBean中包含这么多数据呢？productList就不说了，都知道。

在前端，我们需要知道当前页数，不但可以对当前页数的按钮进行高亮，而且可以判断“上一页”的按钮是否可以使用（当当前页为第一页时，上一页理应不可用）

在前端，我们需要知道总页数，这样才可以在页面中生成各个页的按钮。

其他数据都是相关数据，有可能在前端需要，有些也可能在后端用得上。

根据此bean，我们可以把分页原理很好的进行实现与完善。
