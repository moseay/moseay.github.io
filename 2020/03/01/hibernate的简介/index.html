针对hibernate学习中遇到的一些需要记住的知识点

配置文件

ORM(Object Relation Mapping)配置文件

实体关系映射文件，在hibernate中，我们运用面向对象思维进行数据库操作，在其中一切数据库操作皆为对象操作。因此，我们需要配置对象与表之间的关系。

一般配置文件命名为：实体类简名.hbm.xml

配置文件如下：

<?xml version="1.0" encoding="utf-8" ?>
<!DOCTYPE hibernate-mapping PUBLIC
        "-//Hibernate/Hibernate Mapping DTD 3.0//EN"
        "http://www.hibernate.org/dtd/hibernate-mapping-3.0.dtd">
<hibernate-mapping package="com.ilshmos.domain">
    <class name="Customer" table="cst_customer">
        <id name="cust_id">
            <generator class="native"></generator>
        </id>
        <property name="cust_name"></property>
        <property name="cust_source"></property>
        <property name="cust_industry"></property>
        <property name="cust_level"></property>
        <property name="cust_linkman"></property>
        <property name="cust_phone"></property>
        <property name="cust_mobile"></property>
    </class>
</hibernate-mapping>

这是一个最简单的配置文件，其中，我们首先配置了根元素<hibernate-mapping package="com.moseay.domain">。在这个根元素中，假设我们指定了package属性，填入了实体类所在的包的全包名，那么在下面配置class节点时（或是任意在根元素内部进行实体类名填写时）都可以省略全包名。

<class name="Customer" table="cst_customer">该元素表示了实体与表之间的映射，name属性是实体类的全名，table属性是数据表的名称。

<id name="cust_id">元素代表实体的主键，为何表的字段中要单独列出该元素，会在后面的实体创建规则中讲解。

<generator class="native"></generator>主键生成规则，该属性我们会单独开辟小标题来讲。

<property name="cust_name"></property>剩下的都是实体表中的非主键字段。

id与property元素的属性：

  • name属性代表其实体类对应的成员变量名称。
  • column(可选): 填写表中的主键列名.默认值:列名会默认使用属性名
  • type(可选):填写列(属性)的类型.hibernate会自动检测实体的属性类型.每个类型有三种填法: java类型|hibernate类型|数据库类型
  • not-null(可选):配置该属性(列)是否不能为空. 默认值:false
  • length(可选):配置数据库中列的长度. 默认值:使用数据库类型的最大长度

主键生成规则

  • 自然主键与代理主键：自然主键是实体的业务属性中，包含一个可以作为主键的属性，例如不能相同的用户名；代理主键是指不包含在业务内，手动指定的无关业务意义的主键。

  • 主键生成策略：

    1. identity —— 主键自增，由数据库维护

    2. sequence —— oracle数据库使用

    3. native —— identity，sequence，hilo三选一。（由于几乎市面上的数据库都至少能够支持identity或sequence，所以其实是二选一）

    4. uuid —— 生成uuid作为主键，类型必须是String

    5. assigned —— 自然主键生成，即不做任何操作。由程序员在写业务时自己指定。

    1. hilo —— 本身是主键生成算法，在此代表hibernate实现的主键生成策略。即在插入数据库时，使用它算出主键后填入。不建议使用。
    2. increment —— 主键自增，hibernate自行实现，每次插入时，会先查询最大主键，然后＋1后在执行插入。线程不安全，不建议使用。

主配置文件

主配置文件用于建立hibernate与数据库之间的关系

一般配置默认放在src下，且配置文件名称为：hibernate.cfg.xml

配置文件如下：

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE hibernate-configuration PUBLIC
    "-//Hibernate/Hibernate Configuration DTD 3.0//EN"
    "http://www.hibernate.org/dtd/hibernate-configuration-3.0.dtd">
<hibernate-configuration>
    <session-factory>
         <!-- 数据库驱动 -->
        <property name="hibernate.connection.driver_class">com.mysql.jdbc.Driver</property>
         <!-- 数据库url -->
        <property name="hibernate.connection.url">jdbc:mysql:///hibernate</property>
         <!-- 数据库连接用户名 -->
        <property name="hibernate.connection.username">root</property>
         <!-- 数据库连接密码 -->
        <property name="hibernate.connection.password">root</property>
        <!-- 数据库方言
            不同的数据库中,sql语法略有区别. 指定方言可以让hibernate框架在生成sql语句时.针对数据库的方言生成.
            sql99标准: DDL 定义语言  库表的增删改查
                      DCL 控制语言  事务 权限
                      DML 操纵语言  增删改查
            注意: MYSQL在选择方言时,请选择最短的方言.指定长方言不利于换数据库引擎操作。短方言默认包含其他几种。
         -->
        <property name="hibernate.dialect">org.hibernate.dialect.MySQLDialect</property>
        <!-- 将hibernate生成的sql语句打印到控制台 -->
        <property name="hibernate.show_sql">true</property>
        <!-- 将hibernate生成的sql语句格式化(语法缩进) -->
        <property name="hibernate.format_sql">true</property>
        <!-- 
        ## auto schema export  自动导出表结构. 自动建表
        #hibernate.hbm2ddl.auto create        自动建表.每次框架运行都会创建新的表.以前表将会被覆盖,表数据会丢失.(开发环境中测试使用)
        #hibernate.hbm2ddl.auto create-drop 自动建表.每次框架运行结束都会将所有表删除.(开发环境中测试使用)
        #hibernate.hbm2ddl.auto update(推荐使用) 自动生成表.如果已经存在不会再生成.如果表有变动.自动更新表(不会删除任何数据).
        #hibernate.hbm2ddl.auto validate    校验.不自动生成表.每次启动会校验数据库中表是否正确.校验失败.
         -->
        <property name="hibernate.hbm2ddl.auto">update</property>
        <!-- 引入orm元数据
            路径书写: 填写src下的路径
         -->
        <mapping resource="cn/moseay/domain/Customer.hbm.xml" />
    </session-factory>
</hibernate-configuration>

实体创建规则

实体创建规则：

  1. 无参构造
  2. private成员变量，提供get，set方法
  1. 使用包装类型而非基本类型（针对hibernate）
  2. hibernate操作数据库，数据表必须有主键
  3. hibernate使用cglib代理生成代理对象，因此实体类不能使用final修饰（被final修饰不能被继承，无法创建代理）

注意：hibernate操作数据表，要求数据表必须有主键。因此主键在hibernate中算是一个特殊的数据库字段。

API

Configuration cfg = new Configuration();创建配置类

cfg.configure()默认读取src下的hibernate.cfg.xml进行配置。

cfg.buildSessionFactory()获取session工厂，session是实体与数据库之间交互的“会话”

sessionFactory.openSession();获取新的session

sessionFactory.getCurrentSession()获取当前的session

session.beginTransaction();获取并开启事务

transaction.commit();transaction.rollback();事务的提交与回滚

session.get();session.save();session.update();session.delete();session.saveOrUpdate();session.close();实体状态的修改（并非增删改查等）

实体的三种状态

  • 瞬时状态：无id，不在session一级缓存中
  • 持久化状态：有id，在session一级缓存中
  • 游离状态：有id，不在session一级缓存中

三种状态之间的转换：

get()方法获得的对象为持久化对象。

save()方法将瞬时对象转换为持久化对象。

update()方法将游离对象转换为持久化对象。

delete()方法将持久化对象转换为瞬时对象。

saveOrUpdate()方法是无脑将其他两种状态的对象转换为持久化对象。

close()方法执行后，原本与session关联的持久化对象变为游离对象。

持久化状态的特性：

如果要使数据持久化到数据库，必须将对象转换为持久化状态。

例如，我们通过get()方法获取到一个数据库中的对象，它在获取到时，肯定既包含了ID(数据库中主键非空)，也在session一级缓存中(因为是通过session.get()方法获取到的)。因此它是个持久化对象，那么当我们对对象进行任何修改后。那么它在事务提交操作后会自动持久化到数据库。

hibernate一级缓存测试（session缓存）

public class FirstCacheTest {
    /**
     * hibernate一级缓存测试。
     */
    //证明持久化对象是否就是在session缓存中的对象。
    @Test
    public void confirm(){
        Session session = HibernateUtils.openSession();
        Transaction transaction = session.beginTransaction();
        Customer customer = new Customer();
        customer.setCust_id(1l);//创建对象并设置ID，那么它成为游离状态
        session.update(customer);//将其变为持久状态。
        //如果持久状态对象就是在一级缓存中的对象，那么我们在获取id为1的数据时，它将直接从一级缓存中读取。
        Customer customer1 = session.get(Customer.class, 1l);
        transaction.commit();
        //提交事务后，我们只打印出一条更新操作的sql语句。
        //因为缓存中的数据与快照中比对不一致，会自动将缓存中的数据持久化到数据库。（由于根本没有从数据库中获取该对象，所以不会生成快照，所以缓存一定和快照不一致）
        session.close();
    }
    //测试当我们获取到数据库中的某一行时（此时既有缓存又有快照），假如我们直接手动修改数据，那么重新get到的会是缓存还是新数据。
    @Test
    public void editDataBase(){
        Session session = HibernateUtils.openSession();
        Transaction transaction = session.beginTransaction();
        Customer customer = session.get(Customer.class, 1l);
        //此时对数据库本身进行修改
        System.out.println(customer);
        //此时再次获取该对象（从缓存中获取，没有获取到修改的内容）
        Customer isNewOrCache = session.get(Customer.class, 1l);
        System.out.println(isNewOrCache);
        transaction.commit();
        //由于与快照相同，不会更新到数据库中，数据库中保留新的内容。
        session.close();
    }
}

事务

特性与指定隔离级别

ACID，原子性，一致性，隔离性，持久性

隔离级别：读未提交，读已提交，可重复读，串行化

在数据库用一个字节代表隔离级别：0001,0010,0100,1000（即：1,2,4,8）

在hibernate主配置中指定隔离级别：

<property name="hibernate.connection.isolation">4</property>
<!-- 该配置指定隔离级别为可重复读 -->

使用事务

一开始我们在使用dbutils进行事务操作时，是在工具类中创建了threadlocal对象，绑定了connection对象，因此在调用时，它会拿与当前线程绑定的对象。确保了唯一性。

而hibernate也是用了此原理，但是我们只需要使用session.getCurrentSession()，即可获取与当前线程绑定session。

需要注意的是：

  1. 使用与当前线程绑定的session，必须在主配置中添加一项配置。

    <property name="hibernate.current_session_context_class">thread</property>

  2. 使用getCurrentSession()获取到的session在事务提交后不需要手动关闭，否则会抛出异常。

查询语句

我们已经可以通过session的get方法，获取数据库中id为任意的对象。但是它不能涉及到批量查询与更多的条件查询。

当我们要使用批量或条件查询时，hibernate为我们提供了三种方式。

  1. HQL语句（hibernate自创）（多用于多表不复杂查询），原来的sql语句一般是操作表，而HQL语句中，一切都是在操作实体类。具体相关API此处不再赘述。

  2. criteria查询（hibernate自创）（多用于单表条件查询），此种方式彻底抛弃编写任何语句，一切查询方法都被封装为API。具体API不再赘述。

  3. 原生sql查询（多用于复杂业务查询）
