动态代理-tomcat8.5

在[学习日记20.01.21]中，我们学到了如何通过filter解决全局字符的编码问题。

不过，通过装饰者模式去进行方法增强，远不如动态代理来的简单。

动态代理

什么是动态代理？通俗的讲，动态代理就是在使用目标方法前，通过代理进行“中介”，而在中介的过程中，我们可以轻易的在执行目标方法前后进行一些操作。

通过动态代理与filter实现全局字符编码增强

没错，只要是需要全局增强，那么目前来讲绕不过filter。

我们需要通过filter拦截所有的请求，然后对ServletRequest的对象进行增强。此处我们直接写doFilter(req,reps,chain)方法内部的代码——即动态代理代码。

public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) throws ServletException, IOException {
        //1.首先将拦截到的请求转换为HttpServletRequest
        HttpServletRequest request = (HttpServletRequest) req;
        //2.通过Proxy.newProxyInstance创建动态代理
        /**
        * 其中的三个参数分别为，被代理对象的类加载器，被代理对象的所有接口，
        * 一个InvocationHandler接口的实现类
        * 我们在第三个参数中，也就是InvocationHandler接口的实现类中
        * 写入被代理对象的方法，同时可在其中进行方法增强
        * 而Proxy.newProxyInstance在给定参数后
        * 它的返回值即为“被代理对象”的“代理对象”，与被代理对象同一类型
        */
        HttpServletRequest enhanceReq = (HttpServletRequest) Proxy.newProxyInstance(request.getClass().getClassLoader(), request.getClass().getInterfaces(), new InvocationHandler() {
            @Override
            public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
                //3.获取方法名
                String methodName = method.getName();
                //4.判断是否是指定方法
                if ("getParameter".equals(methodName)) {
                    //5.如果是则获取方法执行后的返回值（参数为被代理对象的实例与方法中本来接受的参数，也就是匿名内部类invoke方法的args
                    String param = (String) method.invoke(request, args);
                    System.out.println("filter:"+param);
                    //6.对返回值进行重编码并返回
                    return new String(param.getBytes("iso8859-1"), "UTF-8");
                }
                //7.其他方法一律不修改直接放行
                return method.invoke(request, args);
            }
        });
        //8.放行filter（需要传入代理对象，而非一开始的被代理对象）
        chain.doFilter(enhanceReq, resp);
    }

tomcat8.5

通过上文的方法，我们已经将全局字符编码给弄好了（注意，只增强了一个获取参数的方法，还有好几个没有增强）

但是，运行时就出现了问题。在运行过程中，通过post方式接收到的参数可以通过该方法输出正常的汉字；而通过get方式接受到的参数依旧是乱码。

我真是黑人问号脸，因为这个方法本来就是针对get方式传参的，只不过它同时也对post有效而已。如果只需要针对post，只需要使用request.setCharacterEncoding("UTF-8");即可。

我已经忘记我是怎么一遍遍调试，因为我刚转用idea，所使用的环境不太熟悉，所以我脑子也一团乱。

我其实一开始就想会不会是环境的问题，但是我还是把环境问题放在了最后考虑，而是先思考是不是我的代码有问题。因为动态代理我也是第一次接触。

不过到了最后，我就去网上搜索了tomcat8的新特性，最后终于发现：tomcat8与tomcat7不同，它解决了get方式传值的字符编码问题。

我经过测试发现，在默认不使用任何的处理方式进行中文传参时：

  1. GET方式传参可以直接输出正确的汉字。
  2. POST方式传参则是乱码。

在使用request.setCharacterEncoding("UTF-8");之后：

  1. GET方式传参可以输出正确的汉字。
  2. POST方式传参也可以输出正确的汉字。

这个问题一定要记录以防以后再次出错…
