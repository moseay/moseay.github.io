DBUtils关闭连接的一些问题

2019年12月07日 1.1k 字 大概 4 分钟 javaweb c3p0 dbutils mysql

问题描述

昨天学习了mysql与jdbc事务的一些操作，在之前写好的DBUtils类中进行了一些关于事务处理的补充。然而在实际在测试项目中使用它的时候，却出现了一个错误：当我在jsp页面点击“提交”后，第一次处理便成功了。而在此时后退页面重新提交，程序便报了一个SQLException：you can’t operate a closed connection!

通过这句话，我定位出导致问题出现的代码：

public static void commit() throws SQLException {
        Connection connection = tl.get();
        connection.commit();
        connection.close();
}

在这里，我在最终提交事务的同时，将该连接关闭。因此在jsp提交又后退再重新提交时，在服务端认为这两次提交都是一个线程，因此在取得connection时则会从线程中取出旧的connection。而在第一次提交数据的最后，该connection已经被关闭，第二次使用时便出现了异常。

疑惑点

根据数据库连接池的定义，理论上连接池会重写Connection类的close方法。在connection对象调用close方法时，它会将连接放回连接池而并非销毁。

这里就是让人疑惑的地方，因为连接并没有销毁而是放回了连接池当中。根据定义去推断，在原ThreadLocal中储存的connection在调用了close方法后，它又回到连接池中。而我在第二次调用时，理想情况下，它应该可以直接使用，而不是出现连接已经被关闭这种情况。

解决

通过百度了一些资料后，在加上我自己的（脑补）思考。我勉强给了这种情况一个说法。

（由于无法获得c3p0连接池完整的源码，无法更深层次的去探究）

问题就出在连接关闭后还没有重新获取就使用了。所以错误信息就提示你，你不能在一个已经关闭的连接上操作。
我想，是因为连接关闭后，它的确会返回到数据连接池中，但是需要重新调用dataSource.getConnection()获取才能对连接进行初始化。
意思就是，虽然连接返回了数据连接池，但是没有进行初始化，这个连接依旧是关闭状态，所以才会导致问题的出现。

可以在拿连接时判断一下连接是否被关闭，如果连接被关闭就取连接池中的连接。如果能从连接池中拿到这个刚被关闭的连接，那它会在被取出时由连接池对它进行初始化，这样才能被使用。当然，也不一定拿到这个连接，但是总体来说拿不拿得到它都不会有任何影响。

如果不从连接池中重新获取它，虽然它在数据连接池中，但它永远是关闭状态。

总体来说，它的内存地址没变，它依旧占用那一块内存，只不过它的状态是“关闭”状态，它无法再进行数据库连接。而是用dataSource.getConnection()重新获取时，c3p0连接池会自动对它进行初始化。

以上“说法”是我自己的猜测，也许并非正确的解释。但是在此处使用这种解释并不会对编写程序产生一些问题。

解决方案代码一：

public static void commit() throws SQLException {
        Connection connection = tl.get();
        connection.commit();
        connection.close();
        t1.remove();
}

这种方法是在每次提交后，就将该连接从ThreadLocal中取出，这样就算第二次使用该线程进行操作时，也会重新从数据连接池中获取新的连接。

解决方案代码二：

public static Connection getCurrentConnection() throws SQLException {
        Connection connection = tl.get();
        if(connection == null || connection.isClosed()) {
            connection = getConnection();
            tl.set(connection);
        }
        return connection;
    }

这种方法，在每次获取“与当前线程绑定的连接”时，都会预先判断与这个线程绑定的连接是否被关闭，如果已经被关闭就从数据连接池中获取一个新的连接。

我个人比较推荐第二种方法，因为如果在某种特例情况下，导致事务在提交前，连接就已经关闭，这时再调用该连接时，依旧会出错。因此，在获取连接时就直接判断连接是否被关闭，是最好的解决方案。
